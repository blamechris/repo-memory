# Skeptic's Audit: Agent-Facing Search & Retrieval Efficiency

**Agent**: Skeptic -- cynical systems engineer; cross-references every claim against actual code
**Overall Rating**: 2.0 / 5
**Date**: 2026-06-09

**Method:** read every file on the search path; ran live timings against this repo's own `.repo-memory/cache.db` (164 cached files, all with AST summaries) and a 2,000-entry synthetic DB.

---

## 1. Component Ratings

### `search_by_purpose` (src/tools/search-by-purpose.ts) — **2/5**

It works and it is fast: **0.43ms** at 164 files, **3.17ms** at 2,000 synthetic entries (measured). The pathPrefix scoping (lines 34–46) is carefully done. But it is capped at 2/5 by the project's own invariant: **it can return stale data.** It serves `summary_json` straight from `getAllEntries()` (line 37) with zero hash validation — edit a file's exports, search still returns the old summary until something else re-indexes. Contrast `get_file_summary`, which hashes the file and checks `cached.hash === currentHash` before serving (src/tools/get-file-summary.ts:43–53). Search is the one summary-serving path that skips the check. Secondary faults: substring matching with no word boundaries or stop-word handling — query "where do we rank files for relevance" returned 20/164 files (limit-saturated) because "do" matches inside "dependency"; matching is one-directional ("auth" matches "authentication" but query "authentication" never matches a stored "auth"); and lines 101–107 synchronously INSERT up to `limit` (20) telemetry rows per search, with an `entries.find()` inside the loop.

### `get_related_files` (src/tools/get-related-files.ts) — **2/5**

Freshness is guaranteed the expensive way (full re-read every call: **26–40ms** at 164 files, measured), so it's never stale — but its output is wrong in three measurable ways. (1) **It returns paths that don't exist on disk:** querying `src/cache/store.ts` returned `src/persistence/db.js` and `src/types.js` as top results — phantom paths from ESM import specifiers. The `getPathVariants` hack (lines 146–160) patches the *query* node only, never candidates or traversal. An agent consuming this will Read-fail. (2) **All top-10 results scored exactly 0.325** — the ranking is a constant (see ranking.ts below). (3) The transitive-dependency classification (lines 67–73) is dead code in practice because BFS dead-ends on phantom nodes (verified: `getTransitiveDependencies('src/cache/store.ts', 3)` ≡ direct deps). Plus write amplification: each call rewrites the entire imports table (~36 DELETEs + 534 INSERTs here); WAL grew to **4.1MB after 3 calls** of a read-only query.

### `ranking.ts` (src/cache/ranking.ts) — **1/5**

"5-signal composite ranking" is the claim; **one signal is live in the shipped path.** Evidence: `get-related-files.ts:128–134` never passes `cacheStore`, so `recency` is hardcoded 0.5 (ranking.ts:226); `changeFrequency` is an admitted placeholder 0.5 (ranking.ts:245–246); without a `taskId` (the common case), `exploredFiles` is empty so `dependencyProximity` returns 0 for everything (ranking.ts:124) and `taskContextRelevance` floors at 0.1 (ranking.ts:206). Score collapses to `0.175 + 0.15 × fileTypeRelevance` — every source file ties at exactly 0.325 (confirmed live: distinct scores in a 29-candidate result were {0.325, 0.22, 0.205} — pure file-type buckets, order within buckets arbitrary). Worse conceptually: `dependencyProximity` measures distance to *task-explored files*, never to the file you asked about, and the relationship classification (`imports` vs `same-directory`) is computed but **not a ranking input** — a direct import ranks no higher than a same-directory bystander. The unit-level functions are individually fine and tested; as a ranking system it does not rank.

### Dependency-graph rebuild behavior (get-dependency-graph.ts, graph/dependency-graph.ts) — **2/5**

Three independent failures. (1) **`DependencyGraph.load()` (dependency-graph.ts:16–28) has zero production callers** — only `tests/unit/dependency-graph.test.ts:131`. Both tools rebuild from disk and *re-persist* the whole imports table per call; the table sat at **0 rows** in this repo's own cache.db, proving the persistence is write-only ceremony. (2) The no-path "summary" mode is garbage at the output level: measured `getMostConnected(10)` = `vitest`, `src/persistence/db.js` (phantom), `path`, `fs`, `os`, `node:path`… — **7 of 10 "most connected files" are npm packages, node builtins, or nonexistent paths**, in a ~5,300-token response. (3) The symbol path (`getEdgesBySymbol`, dependency-graph.ts:83–105) reads raw DB rows, which include edges from *deleted* files — orphans are only GC'd at server startup (server.ts:327), so mid-session deletions yield stale symbol results: an invariant violation. Bonus drift: get-dependency-graph.ts:32–34 omits `.kt`/`.kts`/`.java` that get-related-files.ts:40–43 includes — commit ad95e52 added Kotlin/Java support to only one of the two graph consumers, so the two tools see different graphs on JVM repos.

### Persistence layer fit (src/persistence/db.ts, cache/store.ts) — **4/5**

The least guilty party. Clean migrations, WAL mode, `idx_imports_target` already in place (db.ts:62), sane meta/telemetry tables, prepared statements throughout. The imports schema (source/target/specifiers/import_type) is exactly what graph reuse needs — it's the tools that ignore it. Mild dings: summaries as opaque JSON blobs force `getAllEntries()` + 164 `JSON.parse` calls per search (measured harmless at this scale, ~0.4ms), and the module-level singleton (db.ts:8–9) closes/reopens on project switch — fine for a stdio MCP server, would bite a multi-project host.

## 2. Top 5 Findings (claims vs reality)

**F1 — "Persist the dependency graph" is already built, and it's unplugged.** The candidate idea presumes the graph isn't persisted. Reality: every edge *is* persisted (dependency-graph.ts:56–67), the table and index exist (db.ts migration v3), and a `load()` method exists — production code just never calls it. The real gap is ~20 lines: `load()` + hash-gated incremental `updateFile()` for changed files only. Building "graph persistence" as a feature would duplicate working code; the work item is wiring plus staleness checks, not storage.

**F2 — The graph is structurally broken for TypeScript ESM projects — i.e., for this project itself.** `resolveTarget` (imports.ts:10–21) preserves the `.js` extension from import specifiers, creating phantom nodes. Measured consequences: `getDependents('src/cache/store.ts')` on the real key returns `[]`; transitive traversal at any depth equals direct deps (BFS hits phantom `.js` nodes with no outgoing edges); `get_related_files` emits nonexistent paths to agents; `mostConnected` is dominated by non-files. Any improvement layered on this graph (proximity ranking, graph persistence, symbol search) inherits the breakage. **This is the prerequisite fix for every graph-flavored candidate idea.**

**F3 — The performance problem FTS5/embeddings would solve does not exist; the cost problem is tokens, not milliseconds.** Measured: full-scan search is 3.17ms at 2,000 files — the top of the stated scale — and the linear scan is ~250× cheaper than one agent turn's latency. Meanwhile the search response was ~1,050 tokens (full `exports` arrays for 20 limit-saturated hits) and the dep-graph summary ~5,300 tokens of mostly junk nodes. Also, the embedding/FTS quality pitch assumes the corpus is semantic prose; reality, the AST "purpose" field is template output — `"functions: computeFileTypeRelevance, computeRecency, … — Score file type relevance: source=1.0…"` — identifier soup with one doc sentence. BM25 or embeddings over that approximates the existing substring match at far higher complexity.

**F4 — "Cache correctness over performance — never return stale data" is violated on two shipped paths.** (a) `search_by_purpose` serves summaries with no hash check (search-by-purpose.ts:37–57) while sibling tool `get_file_summary` carefully validates (get-file-summary.ts:43–53) — the project's flagship invariant holds on the slow path and fails on the discovery path agents hit first. (b) `get_dependency_graph?symbol` returns edges from deleted files until the next server restart (GC at server.ts:327 only). Both are quiet failures: nothing in the response marks the data as unverified.

**F5 — The read path is a write path, and telemetry inflates its own ROI.** Every `get_related_files`/`get_dependency_graph` call deletes and reinserts the entire imports table (534 rows here; 4.1MB WAL after 3 calls) — per-call DB churn on a pure query, which also means two concurrent agent sessions on one repo would interleave full-table rewrites. Separately, search telemetry logs `lineCount × 10` tokens as `summary_served` *per matched result* (search-by-purpose.ts:104–106) — a 20-hit search books 20 files' worth of "raw tokens" as if the agent would otherwise have read all of them. The token-savings report this feeds will systematically overstate the tool's value, corrupting exactly the ROI calculus (`frequency × savings − turns × overhead`) the project says it uses to gate new features.

## 3. Concrete Recommendations

1. **Fix import-target resolution at extraction time (prerequisite for everything graph-related).** In `extractImports`/`resolveTarget` (imports.ts:10–21), resolve relative specifiers against the actual file set (`.js`→`.ts`/`.tsx`, try `index.*`) and tag bare/builtin specifiers as external. Then delete `getPathVariants` (get-related-files.ts:146–160) and filter externals out of related-files candidates and `mostConnected`. One fix repairs: nonexistent result paths, dependents lookup, transitive traversal, and the junk graph summary.
2. **Wire `load()` + incremental refresh into both graph tools.** Replace the scan-and-reread loop (get-related-files.ts:28–53, get-dependency-graph.ts:30–43) with: `graph.load()`; hash-check scanned files against the `files` table; `updateFile()` only changed/new ones; delete imports rows for vanished sources at the same time (fixes the stale-symbol path without waiting for startup GC). This is the "persist the graph" candidate, correctly scoped as ~a day of wiring, and it keeps the never-stale guarantee.
3. **Make the ranking rank.** Pass `cacheStore` at get-related-files.ts:128 (one line — activates recency). Add the already-computed relationship as a signal (imports/imported-by > transitive > same-directory). Compute proximity from the *query file*, not only task-explored files. Delete `changeFrequency` until it has data behind it — a constant signal is dead weight that dilutes live ones.
4. **Honor the staleness invariant on search.** Hash-validate just the ≤20 matched entries before returning (sub-millisecond for 20 files) or add `possiblyStale: true` per result; pick one, but don't return unverified summaries silently. Fix telemetry to log estimated savings once per query, not per hit.
5. **Reject: FTS5, embeddings, query-result caching, and a new symbol-search tool.** All four fail the project's own ROI test — search is already 3ms, the corpus is identifier soup, result caching saves milliseconds while adding an invalidation surface (instant invariant risk), and symbol lookup already exists via `get_dependency_graph?symbol` + exports matching (fix its staleness instead; ~100 tokens/turn for a new tool buys nothing). **Accept the token-shaping idea in minimal form:** trim `exports` arrays in search responses, drop sub-threshold noise matches (require ≥1 term of length ≥3 matching on a word boundary), and make no-path `get_dependency_graph` return repo files only — that attacks the measured ~1,050/~5,300-token responses, which is where the real cost is.
6. **Hoist the source-extension list into one shared constant** used by both graph tools (currently drifted: Kotlin/Java in one, not the other).

---

## 4. Overall: **2/5**

The persistence layer is solid and the raw speed numbers are fine at the stated 100–2,000-file scale — but the headline features don't do what they claim. The "5-signal ranking" emits a constant (every result tied at 0.325, measured), the "dependency graph" can't traverse past one hop and names `vitest` and `fs` as the repo's most connected files, `get_related_files` hands agents paths that don't exist on disk, and the project's own first invariant — never return stale data — is violated by both the search tool and the symbol path. Meanwhile the candidate improvement list aims at the wrong target: FTS5 and embeddings address a 3-millisecond non-problem with template-generated text, while the genuinely valuable improvement (use the imports table that's already written on every call) is sitting in the codebase as an untested-in-production `load()` method with zero callers. Fix target resolution, plug in the persisted graph, pass `cacheStore`, and hash-check search results — roughly a week of unglamorous wiring — before spending anything on retrieval exotica.
