# Minimalist's Audit: Agent-Facing Search & Retrieval Efficiency

**Agent**: Minimalist -- ruthless engineer who believes the best code is no code; identifies what to cut
**Overall Rating**: 2.0 / 5 (of the maximalist "build all six" plan; rated 2/10 on the agent's own scale)
**Date**: 2026-06-09

**Method:** read the code, then measured against this repo's actual `.repo-memory/cache.db` (164 cached files, 37.9KB total summary JSON, avg 231 bytes/summary) using the built `dist/`. Note: my benchmark calls populated the `imports` table (534 rows) and `telemetry` (292 rows) as a side effect — which is itself a finding.

## Measured Baseline

| Operation | Measured | At 2,000 files (est.) |
|---|---|---|
| `search_by_purpose`, full call (164 entries) | **1.08 ms** | — |
| Raw substring scan, 2,000 synthetic entries × 4 terms × 3 fields | **1.1 ms** | ~1–3 ms incl. SQL read + JSON.parse |
| `get_related_files`, full per-call graph rebuild | **25.6 ms** | ~400–600 ms (extrapolated) |
| `DependencyGraph.load()` from the persisted `imports` table | **0.18 ms** | ~2 ms |

So "how slow is the in-memory scan at 2,000 files, actually?" has an answer: **about one millisecond.** And the "obvious hotspot" rebuild is ~half a second worst case — against an agent turn dominated by seconds of LLM inference. **No latency here is user-perceptible.** Every candidate idea must justify itself on something other than speed.

## 1. Candidate-by-Candidate

### 1a. SQLite FTS5 / BM25 — Necessity: 1/5
The corpus is ~460KB of text at maximum scale. FTS5 exists to make searches fast when linear scan is too slow; the linear scan is 1.1 ms. You would add a virtual table, sync triggers against `files.summary_json` updates (store.ts:36–85), FTS query-syntax escaping (agents will send `"`, `-`, `:` and get syntax errors), and a migration. In exchange for negative one millisecond. The BM25 argument also fails on specifics: term-frequency and document-length normalization are meaningless over one-line purpose strings of near-uniform length. The existing field weighting (purpose ×3, exports ×2, declarations ×1, search-by-purpose.ts:58–80) *is* the relevance model that matters here. **Minimal alternative:** whole-word-match bonus (~5 lines in the existing loop). **Verdict: do not build.**

### 1b. Embeddings / semantic search — Necessity: 1/5
1. **The consumer is an LLM.** Embeddings buy synonym/paraphrase recall — but the agent calling `search_by_purpose` can rephrase and retry for free, and does. The query rewriter you'd be paying for is already in the loop.
2. **The corpus is generated by this same system.** If lexical recall is poor, the cheap fix is richer AST purpose lines (`ast-summarizer.ts` just became the default, commit 39573a1) — improve the indexed text, not the matcher.
3. **Infrastructure cost is wildly disproportionate.** Either API embeddings (network, key, per-index cost — in a *local stdio server whose entire pitch is reducing token/API spend*) or a local model (~100MB+ ONNX runtime, vector storage, invalidation). This project already has native-dependency pain on record. A second native ML stack for a single user is the definition of resume-driven development.
4. **Scale:** brute-force cosine over 2,000 vectors would be fast — but so is the substring scan you already have.

**Verdict: do not build.**

### 1c. Persisted dependency graph — Necessity: 4/5 (but it's a *wiring fix*, not a build)
The one real defect — and the punchline is **the persistence already exists and is dead code**:
- `DependencyGraph.load()` (dependency-graph.ts:16–28) reads the graph from SQLite. Its only caller in the entire repo is a unit test. No production code calls it.
- `get_related_files` (get-related-files.ts:27–53) and `get_dependency_graph` (get-dependency-graph.ts:27–43) re-read every project file and call `updateFile()` per file on every invocation.
- `updateFile()` does a `DELETE` + transactional `INSERT` per file. So **every read-only query rewrites the entire imports table** — a handful of calls left a 4.1MB WAL next to a 151KB database. Before my benchmark, this repo's `imports` table had **0 rows**: nothing in the indexing pipeline ever populates it.
- The duplicated rebuild loops have already drifted: `.kt/.kts/.java` in one tool, not the other — so the two tools see different graphs. Classic cost of duplicated code.

Measured: load is **0.18 ms vs 25.6 ms**, a 140× win, and it removes write-on-read. But be honest about why this matters: not latency (25 ms is nothing). It matters because it **deletes** two divergent 25-line rebuild loops, stops a read tool from churning the DB, and uses a table + method that already shipped. **Net LOC: negative.**

### 1d. Symbol-level search index — Necessity: 1/5
Already exists in three forms: summaries store `exports` and `topLevelDeclarations` and search matches both; the `imports` table stores per-edge `specifiers` and `getEdgesBySymbol()` queries them, already exposed via the `symbol` param. For symbol *definition* lookup the agent has grep and LSP, which are strictly better. A `symbols` table would be a third copy of existing data with its own invalidation. **Verdict: do not build.**

### 1e. Query-result caching — Necessity: 1/5
A cache in front of a 1 ms query saves ~1 ms and introduces the only failure mode this project explicitly forbids. Agents rarely repeat byte-identical queries within a session; across sessions files have changed, so the cache must be invalidated on any file change — i.e., you'd build the hard part of caching to protect a trivial computation. This is a cache for your cache. **Verdict: do not build.**

### 1f. Token-budget-aware response shaping — Necessity: 2/5
The only candidate aimed at the actual scarce resource (tokens, not milliseconds) — but the heavy version (a `maxTokens` param + token estimator + truncation policies) is overkill when blunt `limit` params already exist. The 80/20 cut is response-shape trimming: `search_by_purpose` returns the **full `exports` array per result** — for a barrel/types file that can be dozens of identifiers × 20 results; `matchedOn` is debug metadata of doubtful value. Capping exports at ~5 and dropping `matchedOn` is a 3-line change that captures most of the savings of a "shaping framework." **Verdict: do the 3-line trim, not the framework.**

## 2. Top 5 Things to NOT Build

1. **Embeddings/semantic search.** Single user, 460KB corpus, LLM consumer that self-rephrases, native-dep pain on record, contradicts the token-thrift premise. Worst ROI by an order of magnitude.
2. **FTS5/BM25.** Replaces a measured 1 ms scan with schema, triggers, and query-escaping bugs. BM25 adds nothing over field weights on one-line documents.
3. **Query-result caching.** Saves ~1 ms; risks the one thing the conventions say is non-negotiable. Frequency × savings ≈ tens × 1 ms ≈ zero.
4. **Symbol-level index.** Third copy of data already in `summary_json` and `imports.specifiers`; grep/LSP own this use case.
5. **A new search/retrieval MCP tool of any kind.** Each tool costs ~100 tokens *every turn*. A new tool must save >100 tokens/turn to break even — none of these do. Anything worth doing fits as a param or internal fix.

Bonus cut: **the 5-signal ranking is mostly theater in its only production path.** `rankFiles` is called once, without `cacheStore` — so `recency` is hardcoded 0.5 (ranking.ts:227–232); `changeFrequency` is a hardcoded placeholder (ranking.ts:246); without a `taskId`, `dependencyProximity` returns 0 (ranking.ts:124) and `taskContextRelevance` is constant 0.1. Default-path ranking reduces to "is it a source file" plus constants. Before adding any new ranking signal, delete or wire the three dead ones.

## 3. The Single Highest-Leverage Minimal Change

**Populate `imports` during summarization, and make the two graph tools call `load()` instead of rebuilding.** Concretely: the summarize/write path already holds the file's contents at exactly the moment a file is new or changed — add the graph update there. Then replace the ~25-line scan-and-rebuild loops in both tools with `graph.load()`.

Why this one: (a) it's the only candidate addressing a defect rather than a hypothetical — read tools currently rewrite the whole table per call and the two tools disagree about which languages exist; (b) it reuses code that already shipped — you are deleting ~50 lines and adding ~5; (c) freshness comes free from the existing hash-based change detection: summaries regenerate when hashes change, so import edges regenerate at the same moment, satisfying "never return stale data" with no new invalidation machinery (edge case: deleted files — hook `deleteEntry`/GC to also delete imports rows). The 140× latency win is real but incidental; the point is one graph, one writer, zero duplication.

Runner-up (same afternoon): cap `exports` at 5 per result and drop `matchedOn` in `search_by_purpose` responses — the only change on the list that saves the resource that actually costs money (tokens).

## 4. Overall Rating of the Maximalist Plan

**Rating: 2/10.** The maximalist plan would spend weeks building FTS5 sync triggers, an embedding pipeline, a symbols table, a query cache, and a token-budgeting framework to optimize a search path whose worst measured cost is one millisecond and whose worst extrapolated cost is half a second, in a tool used by one person, called tens of times per session, where each added tool also bleeds ~100 tokens per turn forever. Five of the six ideas fail the project's own ROI formula on the "frequency × savings" side (savings ≈ 0) while adding real costs on the overhead side; the sixth — the persisted graph — is the exception precisely because it isn't a build at all: the persistence layer already exists, dead, eleven lines long, waiting for a caller. Wire `load()` into the graph tools, populate `imports` from the summarize path, trim the search response shape, and delete the three constant ranking signals — net negative lines of code, and the system gets faster, cleaner, and cheaper in tokens simultaneously. Everything else on the list is solving problems this project has only read about.
